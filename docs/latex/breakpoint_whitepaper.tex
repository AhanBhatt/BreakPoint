\section{Executive Summary}

BreakPoint is an eval data compiler for LLM systems. Its purpose is not to publish another broad leaderboard or hand-write a static set of benchmark prompts. Its purpose is to take real observed failures and compile them into durable, versioned regression data that can be replayed against future prompts, models, RAG pipelines, and agent tool policies.

The core thesis is simple:

\begin{quote}
\textbf{Give BreakPoint one real LLM failure trace, and it creates a failure neighborhood: a base eval case, expected answer, hidden traps, grading rubric, adversarial variants, validation reports, and export artifacts.}
\end{quote}

This makes BreakPoint closer to a compiler or fuzzing system than a normal benchmark. The input is a failure trace or declarative failure specification. The output is a validated regression suite with lineage and export formats.

The current system includes:

\begin{itemize}
  \item A Pydantic data model for eval items, expected answers, traps, rubrics, variants, votes, validation reports, suites, runs, reviews, clusters, and gates.
  \item A failure taxonomy covering hallucination, instruction conflict, multi-hop reasoning, tool misuse, long-context retrieval, refusal boundary, format violation, RAG contradiction, and prompt injection.

  \item Trace2Eval ingestion and compilation from normalized production failure traces.
  \item Mutators that add irrelevant context, reorder facts, rename entities, paraphrase instructions, inject prompt attacks, and add stale contradictory evidence.
  \item Live judge adapters for OpenAI, Anthropic, Gemini, and an OpenAI-compatible vLLM endpoint.
  \item Judge calibration reports, model-under-test comparisons, and CI regression-pack selection.
  \item Exports for JSONL, Hugging Face style rows, DuckDB, OpenAI Evals style YAML, lm-eval YAML/JSONL, CI JSON/Markdown, and FastAPI.
\end{itemize}

\section{Why BreakPoint Exists}

LLM applications fail in ways that are more specific than broad academic benchmark scores can describe. A production RAG system might cite an outdated cached summary instead of a fresh policy. A tool-using agent might trust stale tool output over a live API. A support bot might follow a prompt injection hidden in retrieved context. A safety layer might refuse an allowed educational question or answer a high-stakes individualized one.

Traditional eval creation often starts with humans writing prompts by hand. That does not scale well across real incidents, evolving product behavior, and adversarial variants. BreakPoint instead treats failures as source material. It compiles a failure into many related tests that preserve the original causal structure.

\subsection{The Product Surface}

BreakPoint still exposes familiar evaluation primitives:

\begin{itemize}
  \item projects and dataset versions;
  \item eval suites, cases, checks, and runs;
  \item model outputs and judgments;
  \item human review records;
  \item failure clusters and regression gates;
  \item import, export, CLI, and SDK workflows.
\end{itemize}

The difference is the core. BreakPoint is not primarily a leaderboard. It is a failure-to-eval compiler.

\section{Architecture}

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{artifacts/images/architecture.png}
\caption{BreakPoint turns failure traces into specs, cases, mutations, validation reports, and regression packs.}
\end{figure}

The architecture has six main stages.

\begin{enumerate}
  \item \textbf{Failure traces.} Inputs come from production logs, RAG traces, tool-call traces, support tickets, feedback, red-team transcripts, and incident reports.
  \item \textbf{BreakPointSpec.} The compiler turns traces into declarative failure specs containing contracts, evidence rules, traps, mutators, oracles, and rubrics.
  \item \textbf{Case compiler.} Each spec becomes a concrete eval item with task, context, expected answer, hidden traps, rubric, and metadata.
  \item \textbf{Mutation engine.} Mutators create adversarial variants around the base failure by changing ordering, names, irrelevant context, phrasing, and contradictory evidence.
  \item \textbf{Judge validation.} Local checks and live judges verify that generated cases are clear, answerable, trapped, and formatted correctly.
  \item \textbf{Regression packs.} Accepted cases are exported into JSONL, OpenAI Evals, lm-eval, DuckDB, CI reports, and risk-focused regression packs.
\end{enumerate}

\section{Core Data Model}

BreakPoint keeps compiler metadata attached to every accepted case. This is important because a regression test is more useful when it explains where it came from and how it should fail.

\begin{lstlisting}[language=Python,caption={Simplified eval case shape}]
EvalSuite
  EvalCase
    original_failure_seed
    prompt / context / files / tools
    expected_answer
    hidden_traps
    rubric
    oracle
    mutation_lineage
    validation_report
    failure_family
\end{lstlisting}

Important objects include:

\begin{itemize}
  \item \textbf{RawFailureTrace:} normalized source record with task, model output, expected behavior, retrieved documents, tool calls, feedback, severity, and metadata.
  \item \textbf{FailureSeed:} classified failure family plus the generated BreakPointSpec draft.
  \item \textbf{EvalItem:} task, prompt, context, expected answer, hidden traps, rubric, difficulty, variants, and metadata.
  \item \textbf{EvalCase:} product-layer wrapper around an eval item with lineage and validation state.
  \item \textbf{ValidationReport:} judge votes plus deterministic scores for ambiguity, answerability, trap coverage, and format.
  \item \textbf{RegressionPack:} selected set of case IDs for smoke tests, recent failures, mutations, safety tests, or changed-file focus.
\end{itemize}

\section{BreakPointSpec DSL}

BreakPointSpec is the declarative source form for a failure family. It is intentionally close to infrastructure-as-code: versioned, reviewable, reproducible, and suitable for pull requests.

\begin{lstlisting}[caption={Example BreakPointSpec fragment}]
id: rag.freshness_contradiction.v1
name: RAG freshness contradiction
risk: high

contract:
  answer_policy: Use the newest reliable source.
  require_citations: true
  abstain_when_evidence_missing: true

evidence_rules:
  source_priority:
    - official_policy
    - internal_memo
    - cached_summary
  freshness_field: effective_date

traps:
  - id: stale_source_first
    description: Older source appears before newer authoritative source.
    expected_behavior: Ignore stale source and cite newer source.

mutators:
  - type: reorder_context
  - type: add_irrelevant_context
  - type: inject_prompt_attack
  - type: rename_entities
  - type: contradict_with_stale_source

oracle:
  type: trace_grounded_answer
  required_claims:
    - answer uses newest effective date
  forbidden_claims:
    - answer cites superseded policy
\end{lstlisting}

The compiler uses this specification to create the task, context, expected answer, hidden traps, rubric, and variants. The same spec can be regenerated after changing mutator count, judge policy, or export target.

\section{Trace2Eval}

Trace2Eval is the path from raw production failure to eval seed.

\begin{enumerate}
  \item Load normalized failure traces or ingest source-specific payloads.
  \item Redact secrets and basic PII patterns.
  \item Classify the failure family using trace text, retrieved evidence, tool calls, feedback, and severity.
  \item Generate a BreakPointSpec draft.
  \item Compile the spec into an eval item.
  \item Validate the item with local and optional live judges.
  \item Wrap accepted items into product-layer cases and regression packs.
\end{enumerate}

Supported ingestion adapters include:

\begin{itemize}
  \item OpenTelemetry spans;
  \item OpenInference spans;
  \item LangSmith traces;
  \item LiteLLM logs;
  \item retrieval logs;
  \item support tickets;
  \item user feedback and thumbs-down records;
  \item red-team transcripts;
  \item incident reports.
\end{itemize}

\section{Mutation Engine}

The mutation engine creates a failure neighborhood around each seed. This matters because fixing one exact prompt is not the same as fixing the underlying behavior.

Examples:

\begin{itemize}
  \item \textbf{Reorder context:} move misleading evidence before authoritative evidence.
  \item \textbf{Add irrelevant context:} test whether the model stays focused.
  \item \textbf{Rename entities:} check whether the behavior generalizes beyond exact names.
  \item \textbf{Paraphrase instructions:} avoid overfitting to one phrasing.
  \item \textbf{Inject prompt attack:} place malicious instructions inside retrieved text.
  \item \textbf{Contradict with stale source:} introduce plausible but superseded evidence.
\end{itemize}

In the latest actual run, 12 seed traces produced 36 adversarial variants and 48 total eval cases.

\section{How Live Judges Work}

A live judge is an actual model endpoint used as an eval-data validator. It is not the model being benchmarked. Its job is to judge whether a generated eval case is clear, answerable, grounded, trapped correctly, and formatted well enough to keep.

Current live judge adapters:

\begin{itemize}
  \item \textbf{OpenAI:} Responses API with model \code{gpt-5.4-mini}.
  \item \textbf{Anthropic:} Messages API with model \code{claude-sonnet-5}.
  \item \textbf{Gemini:} \code{generateContent} with model \code{gemini-2.5-flash}.
  \item \textbf{Local vLLM:} OpenAI-compatible chat-completions endpoint with model \code{breakpoint-local-judge}.
\end{itemize}

Each judge receives a compact grading prompt:

\begin{lstlisting}[caption={Conceptual judge input}]
You are grading a generated LLM evaluation item.
Return compact valid JSON only with keys: passed, confidence, rationale.

Task: ...
Context: ...
Expected: ...
Hidden traps: ...
\end{lstlisting}

Each judge must return:

\begin{lstlisting}[caption={Judge output contract}]
{
  "passed": true,
  "confidence": 0.94,
  "rationale": "Clear task, answerable context, expected answer, and trap are present."
}
\end{lstlisting}

BreakPoint parses this into a \code{ModelVote}. The \code{ValidationReport} stores all votes plus deterministic scores.

\subsection{Validation Gate}

A generated case passes validation when:

\begin{itemize}
  \item at least two judges pass it;
  \item ambiguity score is at most 0.32;
  \item answerability score is at least 0.72;
  \item trap coverage score is at least 0.65;
  \item format score is at least 0.80.
\end{itemize}

The deterministic scores make judge behavior less arbitrary. Live judges provide model-based scrutiny; local checks enforce structural quality.

\subsection{Current Scoring Logic}

Current model-output scoring is deterministic:

\begin{itemize}
  \item fail if the output repeats forbidden stale or false claims;
  \item fail if the model refuses an answerable case;
  \item score based on required claim coverage;
  \item pass when required coverage is high enough and no forbidden claim appears.
\end{itemize}

The next step is to add real model/app adapters and optionally route candidate outputs to calibrated live judges using each case rubric.

\section{Security, Cost, and Operational Notes}

\subsection{Credentials}

API keys live in \code{.env}, which is ignored by Git. The source repository includes \code{.env.example} only. Live judging is opt-in through CLI flags, SDK options, or \code{BREAKPOINT\_EXTERNAL\_JUDGES=1}.

\subsection{Cost Guard}

Live judge runs estimate cost before execution. The actual-data command accepts \code{--max-live-cost-usd}. If the estimate exceeds the guard, the command fails before making paid calls.

\subsection{Redaction}

The trace layer performs basic redaction for API-key-like strings, token/secret assignments, emails, and SSN-like patterns. Production deployments should add source-specific allow lists, block lists, and retention policies.

\section{Limitations}

The current system is a strong SDK/CLI implementation, but several items remain before it should be used as a fully automated production gate:

\begin{itemize}
  \item Current calibration labels are seeded from accepted validation rather than independent human labels.
  \item Model-under-test runs are currently simulated profiles; real app/model endpoint adapters are the next major step.
  \item Live judges validate eval item quality; they are not yet the primary oracle for scoring arbitrary candidate model outputs.
  \item Source-specific ingestion has adapters, but real private connectors need deployment-specific authentication, retention, and schema mapping.
  \item CI pack selection exists, but a repository-specific GitHub Action comment bot and historical baseline store should be added.
\end{itemize}

\section{Roadmap}

The next high-value work is:

\begin{enumerate}
  \item Replace provisional gold labels with human reviewer labels and reviewer agreement metrics.
  \item Connect ingestion adapters to real private production systems.
  \item Add real model-under-test adapters for app endpoints, model APIs, and agent traces.
